# ThisOne 질문 이해·추가 조사 레이어

## 1. 왜 이 레이어가 필요한가

ThisOne은 사용자의 원문 질문을 곧바로 검색어로 쓰면 질문을 너무 문자 그대로 해석할 수 있다. 사용자는 늘 검색에 적합한 표현으로 묻지 않으며, 짧은 말·구어체·생략된 맥락·기관명과 역할 단어가 섞인 질문이 자주 들어온다.

따라서 즉답 또는 각 모드의 최종 답변을 만들기 전에 공통 레이어가 먼저 질문을 구조화한다. 이 레이어는 답변을 생성하지 않고, 모드별 답변기가 더 좋은 검색과 근거 확인을 할 수 있도록 분석 결과를 제공한다.

핵심 원칙은 다음과 같다.

> 질문이 좋아야 답이 좋아진다.

> 디스원은 사용자의 원문 질문을 그대로 검색하지 않는다. 먼저 사용자의 의도를 파악해 더 정확한 내부 질문으로 재작성한 뒤, 그 질문을 기준으로 검색·출처 확인·답변을 수행한다.

## 2. ThisOne 처리 흐름

```text
raw user input
→ intent analysis
→ internal rewritten question
→ evidence preference
→ search query generation
→ deeper research if needed
→ interim report
→ mode-specific answer
```

각 단계의 역할은 다음과 같다.

1. **raw user input**: 사용자의 원문 질문을 받는다.
2. **intent analysis**: 질문이 소속/지위 확인인지, 방법 안내인지, 추천인지, 오류 해결인지 같은 넓은 작업 유형을 판정한다.
3. **internal rewritten question**: 검색과 출처 확인에 더 적합한 내부 질문으로 바꾼다.
4. **evidence preference**: 공식 출처, 일반 웹, 사용자 제공 자료, 일반 지식 중 어떤 근거가 우선인지 정한다.
5. **search query generation**: 원문 문장 하나가 아니라 키워드형 검색어 여러 개를 만든다.
6. **deeper research if needed**: 첫 검색에서 충분한 출처가 없으면 즉시 답변을 끝내지 않고 더 깊은 조사 쿼리로 전환한다.
7. **interim report**: 추가 조사로 넘어갈 때만 짧은 한국어 상태 메시지를 제공한다.
8. **mode-specific answer**: 즉답, 해석, 럽미, 집밥, 쇼핑, 서치 등 각 모드가 자기 톤과 정책에 맞게 최종 답변을 만든다.

## 3. 공통 분석 결과

`lib/questionUnderstanding.js`의 `analyzeQuestion(input, options)`는 규칙 기반으로 다음 정보를 반환한다.

- 원문과 정규화된 질문
- 언어 판정
- 의도 유형과 작업 유형
- 내부 재작성 질문
- 엔티티, 핵심 문구, 역할 단어, 기관 단어
- 검색 필요 여부, 최신 정보 필요 여부, 공식 출처 필요 여부
- 근거 선호도
- 1차 검색 쿼리와 추가 조사 쿼리
- 답변 전략
- 추가 조사 전환 시 보여줄 짧은 중간보고 메시지

이 레이어는 **AI 모델을 호출하지 않는다.** 이번 단계에서는 규칙 기반 공통 기반만 만들고, 최종 답변 생성은 각 모드에 남긴다.

## 4. 추가 조사 원칙

> 정보 부족은 답변 종료 사유가 아니라 추가 조사 전환 조건이다.

첫 검색에서 유용한 출처가 부족하면 ThisOne은 곧바로 “현재 정보로는 알 수 없습니다.”라고 끝내지 않는다. 대신 다음을 수행한다.

1. 질문을 더 정확한 검색 질문으로 다시 정리한다.
2. 공식·공공·법령·기관 자료가 필요한 질문이면 공식 출처 중심으로 다시 찾는다.
3. 역할/소속/권한/대상/기한 등 확인되지 않은 부분을 분리한다.
4. 그래도 공개 출처가 충분하지 않으면 확인된 범위, 합리적으로 추정 가능한 범위, 사용자가 다음에 확인할 지점을 나눠서 답한다.

추가 조사 전환 시 사용할 수 있는 짧은 메시지는 다음과 같다.

- 질문 의도를 파악하고 있습니다.
- 검색 질문을 다시 정리하고 있습니다.
- 공식 출처와 관련 자료를 확인하고 있습니다.
- 근거가 부족한 부분을 분리해 정리하고 있습니다.

이 메시지는 정상적인 모든 검색에 반복 노출하지 않고, 1차 검색에서 근거가 약해 추가 조사로 넘어갈 때만 사용한다.

## 5. 이후 적용 대상 모드

이 레이어는 다음 모드가 순차적으로 공유할 수 있다.

- 즉답
- 해석
- 럽미
- 집밥
- 쇼핑
- 서치

이번 PR에서는 **즉답만 최소 통합**한다. 해석, 럽미, 집밥, 쇼핑, 서치의 최종 답변 방식과 톤은 변경하지 않는다.

## 6. 금지 사항

이 문제를 수백만 개의 주제별 특수 처리로 해결하면 안 된다.

- 특정 답변을 하드코딩하지 않는다.
- 특정 한 주제만 겨냥한 예외 규칙을 만들지 않는다.
- 가짜 출처를 만들지 않는다.
- 원문 Serper JSON을 사용자에게 노출하지 않는다.
- 쇼핑 랭킹, 해석 동작, 럽미 톤, 집밥 안전 규칙, analytics, KV는 이 레이어 도입과 별개로 유지한다.

질문 이해 레이어는 계속 범용 규칙이어야 하며, 모드별 답변기는 이 구조화 결과를 받아 각자의 책임 범위에서 답한다.

## 7. 즉답 조사 전략 레이어

`lib/researchStrategy.js`의 `planResearch(analysis, sources)`는 즉답 모드가 1차 검색을 마친 뒤 다음 행동을 정하는 작은 공유 헬퍼다. 질문 이해 레이어가 만든 `needsOfficialSource`, `deeperResearchQueries`와 1차 검색 출처 목록을 함께 보고 아래 값을 반환한다.

- `sourceQuality`: `good`, `weak`, `none` 중 하나로 출처 품질을 표시한다.
- `shouldEscalate`: 출처가 약하거나 없으면 `true`가 되어 추가 조사 검색으로 넘어간다.
- `reason`: 왜 그대로 답하거나 추가 조사가 필요한지 내부 판단 근거를 남긴다.
- `nextQueries`: 추가 조사 시 사용할 `analysis.deeperResearchQueries` 기반 검색어다.
- `answerSections`: 충분한 근거가 있을 때와 약한 근거일 때의 답변 섹션을 구분한다.

판정 원칙은 다음과 같다.

1. 출처가 충분하면 추가 검색하지 않고 `결론`, `근거`, `지금 할 일`, `주의할 점` 구조를 권장한다.
2. 출처가 약하거나 없으면 추가 검색을 시도하고, 그래도 약하면 `결론`, `확인된 내용`, `합리적 해석`, `확인되지 않은 부분`, `지금 확인할 것`처럼 확인 사실과 해석을 분리한다.
3. `analysis.needsOfficialSource`가 `true`인 질문은 공식·공공 도메인을 우선한다. 블로그·카페·포럼류 출처만으로는 좋은 근거로 보지 않는다.
4. 이 레이어는 출처를 만들지 않고, 다중 모델 리뷰 조건도 직접 실행하지 않는다. 최종 답변 생성, 선택적 검토, UI는 기존 즉답 흐름에 남아 있다.

즉답 모드는 1차 Serper 검색 직후 `planResearch()`를 호출한다. `shouldEscalate`가 `true`이면 추가 조사 쿼리로 한 번 더 검색하고, 기존 출처와 병합·중복 제거한 뒤 최종 답변 생성에 전달한다. 병합 후에도 근거가 약하면 “공개 출처를 충분히 확인하지 못해 일반적인 기준으로 정리했습니다.”라는 주의 문구와 함께 확인된 내용과 해석을 분리해 답한다.

## 8. 즉답 다중 모델 답변 검토

즉답 모드는 질문 이해 레이어 자체를 변경하지 않고, 그 결과와 조사 전략 결과를 사용해 필요한 경우에만 다중 모델 검토를 수행한다.

기본 원칙은 비용 통제다. 단순하고 저위험인 일반 질문은 기존처럼 1차 답변 생성 흐름만 사용하며, 항상 Gemini와 OpenAI를 모두 호출하지 않는다.

검토가 필요한 경우는 다음과 같다.

- `analysis.resolutionStrategy`가 `multi_model_review`인 경우
- `analysis.resolutionStrategy`가 `hybrid`이고 출처 품질이 약한 경우
- 공식 출처가 필요한 질문인데 출처 품질이 약한 경우
- 1차 검색 뒤 추가 조사로 escalated 되었고 공식 출처 또는 고위험 주제 확인이 필요한 경우
- 법률, 공공기관/소속, 안전, 노동, 의료, 금융, 자격, 벌금·과태료처럼 단정 답변의 위험이 큰 주제인 경우

검토 흐름은 다음 역할 분담을 따른다.

1. **Gemini**: 기존 primary 모델 흐름으로 1차 답변 초안을 만든다.
2. **OpenAI**: `OPENAI_API_KEY`가 있고 검토 조건을 만족할 때만 검토자로 호출되어 누락 의도, 출처 없는 주장, 과도한 단정, 누락된 주의사항, 미확인 지점을 구조화해 점검한다.
3. **ThisOne**: Gemini 초안과 OpenAI 검토 메모를 바탕으로 최종 답변을 합성한다. 검토 메모 자체를 사실로 승격하지 않고, 확인된 내용·합리적 해석·확인되지 않은 부분을 분리한다.

OpenAI 검토 응답은 가능하면 다음 JSON 형태를 사용한다.

```json
{
  "missingIntent": [],
  "unsupportedClaims": [],
  "overconfidenceWarnings": [],
  "unconfirmedPoints": [],
  "suggestedFixes": [],
  "safeToAnswer": true
}
```

JSON 파싱에 실패해도 사용자 요청은 실패시키지 않는다. 해당 내용은 내부 검토 메모로만 취급하고, 원문 JSON이나 내부 오류를 사용자에게 노출하지 않는다. OpenAI 검토 자체가 실패하면 서버 경고 로그만 남기고 Gemini 초안을 출처 주의 문구와 함께 반환한다.

어려운 공공·공식 질문에서 최종 답변은 가능한 한 다음 구조를 선호한다.

1. 결론
2. 확인된 내용
3. 합리적 해석
4. 확인되지 않은 부분
5. 지금 확인할 것

근거가 약한 경우에는 “공개 출처를 충분히 확인하지 못해 일반적인 기준으로 정리했습니다.”라는 문구를 포함한다.
